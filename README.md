# Zeitgeist

Zeitgeist is a public trend dashboard that collects internet signals from verified
APIs, stores them in Postgres, and serves a clean category dashboard from stored
data. It does not call external source APIs during page loads.

The current product is a Django REST Framework backend, a Next.js frontend, and
a scheduled Cloud Run ingestion job on Google Cloud.

## Current Status

Phase 1 is complete. Phase 2 is in progress and already includes the public
frontend/backend deployment, more source coverage, saved preferences, email OTP
verification, password reset, and Gemini-powered category summaries.

Current deployed categories and sources:

| Category | Source |
|---|---|
| Tech | Hacker News, DEV |
| News | New York Times Most Popular |
| Gaming | RAWG |
| Sports | Football-Data |

Current user-facing features:

- Public dashboard and `/category/[slug]` detail pages.
- Stored trend data from Postgres, not direct browser calls to source APIs.
- Source-specific trend cards, including football match cards for Sports.
- Category summaries generated by Gemini during ingestion and stored in Postgres.
- Email/password signup and signin using Django session authentication.
- CSRF-protected browser API calls.
- SMTP one-time-password verification for registration.
- Forgot-password flow using the same OTP pattern.
- Saved category preferences for signed-in users.

## Architecture

| Layer | Technology |
|---|---|
| Backend | Django 5.1 + Django REST Framework |
| Frontend | Next.js App Router + TypeScript |
| Database | Postgres 16 on Cloud SQL |
| Batch ingestion | Cloud Run Job triggered by Cloud Scheduler |
| AI summaries | Vertex AI / Google Gen AI SDK with Gemini |
| Secrets | GCP Secret Manager |
| Images | Artifact Registry |
| Infrastructure | Terraform |
| CI/CD | GitHub Actions with Workload Identity Federation |

Runtime ownership:

- Terraform owns stable infrastructure: Cloud SQL, Cloud Run service/job shells,
  Scheduler, Artifact Registry, Secret Manager containers, IAM, and scaling.
- GitHub Actions CD owns runtime images, migrations, seed data, and Cloud Run
  runtime environment/secret attachment.
- Secret values are never stored in Terraform. Terraform creates secret
  containers only; `infra\secrets.bat` or CD supplies values.

| Secret | Used by |
|---|---|
| `django-secret-key` | Django runtime |
| `db-password` | Cloud SQL connection |
| `nytimes-api-key` | News ingestion |
| `rawg-api-key` | Gaming ingestion |
| `football-data-api-key` | Sports ingestion |
| `smtp-host` | Registration/password-reset email |
| `smtp-host-user` | Registration/password-reset email |
| `smtp-host-password` | Registration/password-reset email |

Deferred or unused sources do not have adapter files, seed mappings, CD wiring,
or Terraform secret containers. New sources must be verified with a live API
fetch before code, secrets, or deployment wiring are added.

## Local Development

Start Postgres:

```cmd
docker compose up -d
```

Set up the backend:

```cmd
cd backend
python -m venv .venv
.venv\Scripts\activate
python -m pip install -r requirements-dev.txt
copy .env.example .env
python manage.py migrate
python manage.py seed_categories
python manage.py runserver localhost:8000
```

Run ingestion locally:

```powershell
cd backend
$env:DJANGO_SETTINGS_MODULE="config.settings.development"
python run_job.py
```

Run the frontend:

```cmd
cd frontend
npm install
npm run dev
```

Local URLs:

- Frontend: `http://localhost:3000`
- Backend health: `http://localhost:8000/api/v1/health/`
- Dashboard API: `http://localhost:8000/api/v1/dashboard/`
- Django admin: `http://localhost:8000/admin/`

For local SMTP testing, fill these in `backend\.env`:

```text
EMAIL_BACKEND=django.core.mail.backends.smtp.EmailBackend
EMAIL_HOST=smtp.example.com
EMAIL_PORT=587
EMAIL_HOST_USER=your-email@example.com
EMAIL_HOST_PASSWORD=replace-me
EMAIL_USE_TLS=true
EMAIL_USE_SSL=false
EMAIL_VERIFICATION_OTP_TTL_MINUTES=10
EMAIL_VERIFICATION_MAX_ATTEMPTS=5
EMAIL_VERIFICATION_RESEND_COOLDOWN_SECONDS=60
AUTH_RATE_LIMIT_WINDOW_SECONDS=60
AUTH_RATE_LIMIT_IP_REQUESTS=30
AUTH_RATE_LIMIT_EMAIL_REQUESTS=10
```

`DEFAULT_FROM_EMAIL` is derived in code as `Zeitgeist <EMAIL_HOST_USER>` unless
explicitly overridden.

## Testing

Backend:

```cmd
cd backend
python manage.py makemigrations --check --dry-run
python -m pytest --no-cov
python -m ruff check .
python -m mypy apps config
```

Frontend:

```cmd
cd frontend
npm run typecheck
npm run lint
npm run build
```

Terraform:

```cmd
cd infra
terraform fmt -check -recursive
terraform validate
terraform plan
```

Do not run `terraform apply` until the plan has been reviewed.

## Cost Controls

Pause the scheduler and stop Cloud SQL when you are not using the demo:

```cmd
gcloud scheduler jobs pause zeitgeist-daily-ingest --location us-central1 --project zeitgeist-499322
gcloud sql instances patch zeitgeist-pg --activation-policy=NEVER --project zeitgeist-499322
```

Restart:

```cmd
gcloud sql instances patch zeitgeist-pg --activation-policy=ALWAYS --project zeitgeist-499322
gcloud scheduler jobs resume zeitgeist-daily-ingest --location us-central1 --project zeitgeist-499322
```

## Source Verification Rule

Before adding a source:

1. Confirm auth/key requirements and free-tier limits.
2. Fetch live sample data.
3. Confirm response fields support a useful UI.
4. Add the adapter, seed mapping, tests, docs, and secrets only after verification.

Deferred sources and alternate auth/email providers are not part of the current
implementation and should not be added back without a live verification pass.
